# Deploying Superset to AWS ECS Fargate

Architecture: **CloudFront → ALB → ECS Fargate (Superset)**, with Cognito OIDC for authentication and Snowflake as the data source.

## Prerequisites

- AWS CLI configured with appropriate permissions
- Node.js 20+ and npm (for CDK)
- Docker (for building the image)
- An existing Cognito User Pool with:
  - An app client (with client secret enabled)
  - A domain configured
  - Groups created: `am-infra-emi-admin`, `am-infra-emi`, `am-non-emi`
  - Callback URL set to `https://<your-cloudfront-domain>/oauth-authorized/cognito`
- An existing CloudFront distribution
- Snowflake account with connection details

## Step 1: Deploy Infrastructure

```bash
cd deployment/cdk
npm install

# Deploy with your Cognito details
npx cdk deploy SupersetInfra

npx cdk deploy SupersetEcs \
  -c cognitoDomain=YOUR_COGNITO_DOMAIN \
  -c cognitoClientId=YOUR_CLIENT_ID \
  -c cognitoUserPoolId=YOUR_POOL_ID \
  -c cognitoRegion=us-east-1 \
  -c supersetPublicUrl=https://superset.yourdomain.com
```

## Step 2: Set Secrets

After the infra stack deploys, set the Cognito client secret:

```bash
aws secretsmanager put-secret-value \
  --secret-id superset/cognito-client-secret \
  --secret-string "YOUR_COGNITO_CLIENT_SECRET"
```

The RDS credentials and Superset SECRET_KEY are auto-generated by CDK.

## Step 3: Initialize the Database

Run the init task once (get the task def ARN and cluster name from CDK outputs):

```bash
aws ecs run-task \
  --cluster SupersetCluster \
  --task-definition <InitTaskDefArn> \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={subnets=[<private-subnet-id>],securityGroups=[<ecs-sg-id>],assignPublicIp=DISABLED}"
```

Monitor the init task in CloudWatch Logs under `/ecs/superset/init`.

## Step 4: Point CloudFront to ALB

1. Get the ALB DNS name from CDK outputs (`AlbDnsName`)
2. In your CloudFront distribution, add/update the origin:
   - Origin domain: `<AlbDnsName>`
   - Protocol: HTTP only (port 80)
   - Origin path: empty
3. Configure behaviors:
   - Cache policy: `CachingDisabled` (or a custom policy that respects cookies)
   - Origin request policy: `AllViewer` (forward all headers, cookies, query strings)
   - Viewer protocol policy: `Redirect HTTP to HTTPS`
4. Add the Cognito callback URL to your Cognito app client:
   - `https://<cloudfront-domain>/oauth-authorized/cognito`

## Step 5: Connect Snowflake (dbt & dbt Elementary)

Superset reads from the tables and views materialized by dbt and dbt Elementary in Snowflake.

1. Log into Superset as admin
2. Go to **Data → Databases → + Database**
3. Select **Snowflake**
4. Enter your connection string:
   ```
   snowflake://{user}:{password}@{account}.{region}/{database}?role={role}&warehouse={warehouse}
   ```
   Or use key-pair authentication via the advanced settings.

### Recommended Snowflake databases to connect

| Database | Purpose |
|----------|---------|
| dbt production database | Tables/views materialized by dbt (marts, staging, intermediate models) |
| dbt Elementary database | dbt Elementary observability tables (test results, model run times, data quality) |

**Tips:**
- Use a **read-only Snowflake role** for the Superset service account
- Point Superset at the **production** dbt target (not dev/CI schemas)
- dbt Elementary tables are typically in an `elementary` schema — add this as a separate dataset source
- Use Superset's **SQL Lab** to explore available schemas before building dashboards

## Updating Superset

```bash
# Rebuild and push image
cd deployment/cdk
npx cdk deploy SupersetEcs

# Run migrations (if needed)
aws ecs run-task --cluster ... --task-definition <InitTaskDefArn> ...
```

## Troubleshooting

### Check service logs
```bash
aws logs tail /ecs/superset/web --follow
aws logs tail /ecs/superset/worker --follow
```

### Shell into a running container
```bash
aws ecs execute-command \
  --cluster SupersetCluster \
  --task <task-id> \
  --container superset-web \
  --interactive \
  --command "/bin/bash"
```

### Common issues

| Issue | Fix |
|-------|-----|
| 502 from ALB | Check health check: `curl http://<alb>/health`. Increase `startPeriod` if Superset is still booting. |
| OIDC redirect loop | Verify Cognito callback URL matches exactly: `https://<domain>/oauth-authorized/cognito` |
| Snowflake connection fails | Ensure ECS security group allows outbound HTTPS (443) to Snowflake endpoints |
| Celery tasks not running | Check worker logs. Verify Redis connectivity from ECS security group. |
| "Invalid SECRET_KEY" | Ensure the `superset/secret-key` secret exists in Secrets Manager |

## Security Notes

- **Not publicly accessible** — ALB security group restricts inbound to CloudFront managed prefix list only (no direct ALB access)
- **Authentication required** — Cognito OIDC enforces login; only users in mapped groups (`am-infra-emi-admin`, `am-infra-emi`, `am-non-emi`) can access Superset
- All secrets are stored in AWS Secrets Manager (never in env files or code)
- RDS and Redis are in isolated subnets with no public access
- ECS tasks run in private subnets with NAT gateway for outbound
- TLS is terminated at CloudFront; ALB uses HTTP internally
- `ENABLE_PROXY_FIX = True` is set to correctly handle `X-Forwarded-*` headers
- Session cookies are `Secure`, `HttpOnly`, and `SameSite=Lax`
